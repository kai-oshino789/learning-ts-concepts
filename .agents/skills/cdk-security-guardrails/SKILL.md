# AWS CDK セキュリティガードレール

AWS CDK を用いてインフラを拡張・変更する際、AIアシスタントは以下のセキュリティルールを厳密に遵守してコードを生成しなければならない。

## 1. S3 バケットの保護
- 新規の S3 バケットを定義する際は、必ずパブリックアクセスを完全ブロックし、SSL通信を強制するプロパティを設定すること。
  - `blockPublicAccess: s3.BlockPublicAccess.BLOCK_ALL`
  - `enforceSSL: true`
- すべてのバケットはデフォルトでサーバーサイド暗号化（できれば KMS CMK）を有効化すること。

## 2. KMS 暗号化の一元化
- 永続的なデータまたはログを保持するリソース（RDS, Secrets Manager, CloudWatch Logs, S3）を新設する際は、必ず KMS カスタマー管理キー（CMK）を指定して暗号化すること。
  - デフォルトの AWS 管理鍵（`aws/s3` や `aws/rds` 等）は極力避け、既存の一元化された CMK を参照して紐付けること。

## 3. ネットワーク ＆ セキュリティグループ
- データベース（RDS）やキャッシュ（Redis）などの内部リソースのセキュリティグループにおいて、ソースに `0.0.0.0/0`（全開放）を指定することは厳禁とする。
- インバウンド制限には、必ずアクセス元のリソース（ECSサービスなど）のセキュリティグループオブジェクトまたは特定の IP CIDR を明示的に指定すること。

## 4. 本番データの削除保護
- 本番環境（`envName === "prod"`）でデータベース（RDS）やストレージ（S3）をデプロイする際は、スタック削除時でもデータが保護されるよう以下を設定すること。
  - `removalPolicy: cdk.RemovalPolicy.RETAIN`
  - データベースの場合はさらに `deletionProtection: true` を明示すること。
