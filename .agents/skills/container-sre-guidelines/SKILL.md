# コンテナセキュリティ ＆ 非破壊的運用設計ガイドライン

コンテナ（Dockerfile/nginx.conf等）およびインフラ（ECS Fargate等）を設計・変更する際、AIアシスタントはシステムのセキュリティと運用の可用性を高めるために以下のルールを厳密に遵守しなければならない。

## 1. コンテナイメージのセキュリティ（非Root実行・バージョン固定）
- **非Root実行化**: Dockerfile 内でアプリケーションや Web サーバー（Nginx等）を起動する際は、特権ユーザー（root）ではなく、非特権ユーザー（例: `nginx`, `node` など）を明示的に使用して実行するように設定すること。
  - ポート `80` のような特権ポートではなく、`8080` などの非特権ポートで待ち受ける構成にすること。
  - PIDファイルの書き込み先やキャッシュディレクトリ等のパーミッションエラーを回避するため、PIDや一時ファイルパスを `/tmp` など非特権ユーザーが書き込める場所に明示的に変更した Nginx/App 設定ファイルを用意すること。
- **ベースイメージの固定**: Dockerfile やタスク定義内のベースイメージタグは `latest` を使用せず、マイナーバージョンまでピン留めされたバージョン（例: `nginx:1.25.4-alpine`）を指定すること。

## 2. 非破壊的・即時運用のためのアーキテクチャ設計
- 計画メンテナンスや緊急のトラフィック制御などを設計する際は、インフラ（CDK/Terraform）の再デプロイやコンテナの再起動（データプレーンの変更）を伴わず、数秒で反映可能な「APIレベル（コントロールプレーン）での切り替え」を最優先で考慮すること。
  - **メンテナンス切り替え**: WAFv2のルールアクションを `Count` ➔ `Block` に変更する仕組みや、Route 53 レコードの切り替えなどで即時サービス制限を行えるように設計する。
  - 503エラーなどのカスタムエラーページは、WebACL や CloudFront 自体に静的コンテンツ（HTML）を事前に埋め込む方式とすること。
