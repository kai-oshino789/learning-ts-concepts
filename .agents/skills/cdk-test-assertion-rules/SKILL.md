# AWS CDK テストコード自動強制ルール

インフラコード（CDK）の拡張や変更を行う際、AIアシスタントはデプロイ前の品質を担保するため、必ず対応する単体テストコードを同期して作成・変更しなければならない。

## 1. テストカバレッジの維持
- `infra/lib/` 配下の Construct や Stack を新規追加または修正した際は、必ず `infra/test/stack.test.ts` にテストケースまたはアサーションを追加すること。
- テストのないリソースを新設することは禁止とする。

## 2. 環境別のテストアサーション（環境パリティ検証）
- テストコードは、`dev` (開発), `stg` (検証), `prod` (本番) のすべての環境のテストケースにおいて、追加リソースが仕様通りに構成されているかを検証すること。
- 特に以下の環境ごとのスペックの作り分け（パリティ）を厳密にアサーションすること：
  - 各環境における期待されるリソース数（例: RDS Proxyがdevでは0、stg/prodでは1など）
  - 各環境の削除ポリシー（removalPolicyがdevではDESTROY、prodではRETAINであること）
  - オートスケーリング設定の有無（devには存在せず、stg/prodには存在すること）

## 3. テストの実行と検証
- コード変更完了後、回答をユーザーに返す前に、ローカル環境で必ずテストスイートを実行し、テストが 100% グリーン（パス）であることを自律的に確認すること。
  - 実行コマンド: `npm test`（または `npx jest`）
