# コスト最適化（FinOps）および環境別構成ルール

AWS インフラ（CDK/Terraform）を設計・拡張する際、AIアシスタントは不要なクラウドコストを削減し、持続可能なリソース配置（FinOps）を行うために以下のルールを厳密に遵守しなければならない。

## 1. 開発環境（dev）のコスト最小化
- `dev` 環境は、動作確認に必要な最小限のスペックおよびコンポーネント数で構築すること。
- **原則シングル構成**: リードレプリカの追加や、高可用性のための複数アベイラビリティゾーンへの冗長化（データベース等）は `dev` では行わない。
- **RDS Proxy の省略**: `dev` 環境では RDS Proxy はプロビジョニングせず、アプリケーションは直接データベースに接続する構成とすること。
- **NAT Gateway の回避**: `dev` 環境のプライベートサブネットから S3 などの AWS サービスに接続する際は、高価な NAT Gateway を経由させるのではなく、料金のかからない VPC ゲートウェイエンドポイント（S3 ゲートウェイエンドポイント等）を優先的に使用すること。

## 2. 夜間自動停止の適用
- `dev` 環境におけるインフラ構築時には、稼働時間外（夜間・休日）の不要な課金を防ぐ自動停止ロジックを必ず組み込むこと。
  - ECS Fargate タスク数を夜間に 0 へ自動スケールダウンするスケジュール設定。
  - Aurora DB クラスターを夜間に一時停止・朝に起動する Lambda 関数と EventBridge ルールの定義。
- `stg` および `prod` 環境には、この夜間一時停止スケジュールを適用しないこと。

## 3. 本番・検証環境（prod / stg）の冗長性と割引の考慮
- `stg` / `prod` 環境においては、可用性と信頼性を最優先とし、必要に応じてリードレプリカや RDS Proxy を構成すること。
- コスト削減の提案を行う際は、単なるスペックダウンだけでなく、Compute Savings Plans（ECS Fargate用）や Reserved DB Instances（Aurora/Elasticache用）を前提とした FinOps 最適化シナリオを提示すること。
